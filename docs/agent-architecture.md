# 代理架構：機制與內容分離的雙層 GOAP

定案日期：2026-07-05（與使用者討論確認）

## 設計原則

1. **程式只內建「機制」，不內建「內容」**。
   怎麼讀 HP 弧、怎麼開敵方資訊面板、怎麼辨識範圍疊加層、怎麼讀遊戲顯示的
   預期傷害——這些是固定機制，寫死在程式裡。任何特定關卡的敵方數值、
   特殊能力、光環效果，一律在執行期從畫面獲取，程式碼裡不出現。
2. **不建關卡資料庫**。跨執行不保存任何關卡的敵方單位數值。執行產生的
   流水帳只作工程分析（除錯、歸因規則調校）用，不回饋成下次執行的先驗。
   我方的戰力/資源由遊戲伺服器天然持久，每次執行開始用 OCR 重新讀取，
   程式自己不存——跨執行完全無狀態。
3. **執行輸入極簡**：每次執行只給兩樣東西——
   - 目標：通關某一關。
   - 可用的局外成長 action 集合（強化、開發、編成、farming……逐版擴充）。
   其餘一切由感知現場取得。
4. **關內動作空間完全開放**。戰鬥中允許按任何遊戲提供的按鈕。控制器從
   目前的固定狀態機逐步演化為「掃描當前畫面的可用行動 → 以通用評估函數
   挑選」，而不是限定在攻擊/移動/待機的封閉選單。

## 知識分層（依生命週期）

| 層 | 生命週期 | 內容 |
|---|---|---|
| 機制層 | 永久（程式碼） | UI 文法：HP 弧偵測、威脅/移動格、預期傷害讀取、敵方資訊面板的開啟與解析、範圍疊加層辨識、OCR、畫面錨點 |
| 執行層 | 單次執行（run-scoped 黑板） | 本關敵方觀測（位置、HP、實測傷害、查到的能力）、戰鬥流水帳、門檻信念、戰敗歸因結果 |
| 遊戲層 | 遊戲伺服器持久 | 我方戰力、資源、機體清單——每次執行重新 OCR，不落地 |

## 雙層 GOAP

### 戰略層

- 狀態述詞每輪重新感知，不依賴記憶：
  `power_sufficient`（OCR 戰力 ≥ 執行內門檻信念）、`can_afford_enhance`、
  `has_sweep_charges`、`stage_cleared`。
- 門檻信念初始值 = 關卡顯示的建議戰力；挑戰失敗在**同一次執行內**上調並
  replan（farm → 強化 → 再挑戰的循環在單次執行內完成），執行結束即丟棄。
- action 集合由執行輸入給定，v1 = {挑戰目標關卡、強化機體、略過掃蕩、
  重打已通關卡}。ChallengeStage 內部就是整條戰術層通關迴圈。

### 戰術層

- **run-scoped 盤面黑板**：戰鬥中持續累積的觀測——各單位位置與 HP 弧、
  每次交戰的（攻擊方、武器、預期傷害、實際傷害）、敵方每回合的移動與
  目標選擇、EN 耗盡等事件。開戰時空白，終戰即封存為流水帳。
- **開放行動掃描**：每個決策點列舉畫面上實際可用的行動（武裝槽、技能、
  SUPPORT、MANEUVER、待機……），不是預設清單。
- **評估策略分級**（依序實作）：
  1. 1-ply 貪婪：讀遊戲顯示的預期傷害/命中選武器×目標——遊戲本身就是
     forward model，零資料成本。
  2. 1.5-ply 站位安全：回合終點位置以威脅格＋黑板裡的敵方實測傷害評估
     「站這裡下回合會挨多少打」。
  3. 更深搜尋（2-ply+）僅在戰後歸因顯示「戰力達標卻輸在戰術」時才投資
     （需要敵方策略模型，成本高、雜訊大，不預先做）。

## 執行期敵情獲取機制

- 敵方單位可點擊 → 資訊面板列出能力與屬性。面板**結構**標定一次（機制），
  **內容**每關現讀。
- 地圖砲/招式範圍以顏色疊加層通用辨識——不需要知道那是誰的什麼招，
  只需要知道「這些格子危險」。
- 交戰實測傷害即時記入黑板，回合內就能用於站位評估；同型敵機共享估計。

## 戰後歸因（觀測式，非模擬式）

流水帳證據 → 診斷 → 補強方向的對映，關鍵是能分辨
「該花資源養成」vs「該修我們的程式」：

| 證據 | 診斷 | 補強 |
|---|---|---|
| 前兩回合我方大量陣亡、擊殺少 | 純數值差距 | farm＋強化（戰略層） |
| 打得中但敵 HP 弧幾乎不掉 | 火力不足 | 武器強化（戰略層） |
| 多次 EN 耗盡只能待機 | 戰術缺陷 | 行動掃描＋技能使用（改程式） |
| 同一單位總是最先死 | 編成短板 | 編成調整（戰略層 v2） |
| 回合耗盡未分勝負 | 機動/索敵問題 | 改程式 |

## 工程評估基準（先實作、以檢核點迭代）

- **E1**：第 12 關重播，零關卡專屬資料通關（現況回歸基準，順帶驗證
  62bef06 的敵我分類與平移偵察）。
- **E2**：HARD 1 探測帶流水帳出戰，產出第一份戰敗歸因報告與戰敗畫面錨點。
- **E3**：戰略迴圈在單次執行內走完 farm → 強化 → 再挑戰。
- **E4**：HARD 1 通關，全程無任何手工關卡資料。

## 實作順序

1. 戰鬥流水帳 + run-scoped 黑板（機制層基礎，同時是歸因證據與未來
   模型的訓練資料格式）。
2. HARD 1 探測（E1+E2：先第 12 關回歸，再 HARD 1 收戰敗流程）。
3. OCR 數字讀取（戰力/資金/預期傷害三處共用同一機制）。
4. 戰後歸因 v1：先分「數值差距 vs 程式缺陷」兩大類。
5. 戰略迴圈：goal + allowed actions 輸入介面，強化與 farming 流程標定。
6. 行動掃描器 v1：發現並使用 SUPPORT/技能鈕（EN 補給、回血）。
7. 1.5-ply 站位安全評估。
