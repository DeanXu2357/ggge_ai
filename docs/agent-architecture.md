# 代理架構：機制與內容分離的雙層 GOAP

定案日期：2026-07-05；2026-07-06 修訂（內層 GOAP、行動空間雙來源、
目標介面預留戰役、編成兩階段、啟動順序規劃、MCTS 非禁令）

## 設計原則

1. **程式只內建「機制」，不內建「內容」**。
   怎麼讀 HP 弧、怎麼開敵方資訊面板、怎麼辨識範圍疊加層、怎麼讀遊戲顯示的
   預期傷害——這些是固定機制，寫死在程式裡。任何特定關卡的敵方數值、
   特殊能力、光環效果，一律在執行期從畫面獲取，程式碼裡不出現。
2. **學到的信念跨執行無狀態；靜態內容可作「感知記憶化」cache**
   （2026-07-05 使用者修訂）。門檻信念、歸因結果等學習產物只活在單一
   process；流水帳檔只作工程分析用。但遊戲寫死不變的內容（各關敵方
   數值/武裝/能力、可取得機體與角色的圖鑑資料）允許 cache 加速開局
   同步——**cache 是上次從畫面讀到的東西，畫面永遠是權威**：未命中或
   校驗失敗一律退回現場讀取，且現場讀取的結果寫回 cache。
   我方單位受養成影響，**實際數值永遠現場讀**，圖鑑 cache 只記身分與
   基礎資料。我方戰力/資源由遊戲伺服器天然持久，每次執行重新 OCR。
3. **執行輸入極簡**：每次執行只給兩樣東西——
   - 目標：通關某一關（v1）。介面預留戰役目標（「通關到 HARD X」等，
     2026-07-06 使用者定案）——戰役進度由遊戲伺服器天然持久（關卡列表
     的通關標記），現場讀取即可，不違反無狀態原則。
   - 可用的局外成長 action 集合（強化、開發、編成、farming……逐版擴充）。
   其餘一切由感知現場取得。
4. **關內動作空間完全開放，且來源有二**（2026-07-06 使用者定案擴充）。
   戰鬥中允許按任何遊戲提供的按鈕，但光靠畫面掃描不夠——擊殺再動這類
   能力在決策當下的畫面上看不到，是編成畫面才讀得到的知識。因此：

   ```
   戰鬥行動空間 = 畫面掃描（機制層，行動掃描器）
                ∪ 能力注入（內容層，戰略層編成時現場讀取後灌入黑板）
   ```

   控制器從固定狀態機演化為「行動目錄（ActionCatalog）→ 規劃器挑選」，
   而不是限定在攻擊/移動/待機的封閉選單。

## 知識分層（依生命週期）

| 層 | 生命週期 | 內容 |
|---|---|---|
| 機制層 | 永久（程式碼） | UI 文法：HP 弧偵測、威脅/移動格、預期傷害讀取、敵方資訊面板的開啟與解析、範圍疊加層辨識、OCR、畫面錨點 |
| 執行層 | 單一 process（run-scoped 黑板） | 該關的通關記憶：敵方能力/光環、實測傷害、門檻信念、歷次嘗試的流水帳與歸因——同一 process 內反覆挑戰會持續更新，process 結束即丟棄 |
| 戰鬥層 | 單場戰鬥（盤面狀態） | 當前各單位位置與 HP、本場事件流水帳——終戰封存進黑板 |
| 遊戲層 | 遊戲伺服器持久 | 我方戰力、資源、機體清單——每次執行重新 OCR，不落地 |

「單次執行」的定義（使用者 2026-07-05 確認）：一次明確給定目標與手段的
執行腳本 process。process 內因無法通關而重複挑戰時，該關難點記憶在黑板中
並隨重複過程更新；process 結束後，下次執行的黑板從零重建。

## 雙層 GOAP

### 戰略層

- 狀態述詞每輪重新感知，不依賴記憶：
  `power_sufficient`（OCR 戰力 ≥ 執行內門檻信念）、`can_afford_enhance`、
  `has_sweep_charges`、`stage_cleared`。
- 門檻信念初始值 = 關卡顯示的建議戰力；挑戰失敗在**同一次執行內**上調並
  replan（farm → 強化 → 再挑戰的循環在單次執行內完成），執行結束即丟棄。
- action 集合由執行輸入給定，v1 = {挑戰目標關卡、強化機體、略過掃蕩、
  重打已通關卡}。ChallengeStage 內部就是整條戰術層通關迴圈。
- **前置條件即內容**（2026-07-06 使用者定案）：action 的前置條件不寫死
  在程式，由資料綁定——
  1. 有現成資料（畫面讀取或內容 cache）：如「挑戰 HARD 3」的前置 =
     `體力 ≥ 該關顯示的出擊消耗`（關卡資訊畫面現讀，一般 10 點）。
  2. 無現成資料：用 LLM 或其他啟發式手段推斷該 action 的前置條件
     （留待 LLM 整合階段）。
  體力管理因此不是特例補丁：體力是現場 OCR 的資源述詞，挑戰 action 掛
  資料綁定的前置；前置不滿足時 planner 自然改走能滿足它的 action
  （等待回復、道具……隨 action 集逐版擴充）。
- **編成 action 分兩階段**（2026-07-06 使用者定案）：
  1. **讀取同步**（挑戰的前置）：逐出擊單位讀能力清單寫入黑板
     `TeamRoster`/`UnitCapability`——這是內層能力注入的唯一來源。
  2. **主動調整編成**：等歸因出現「編成短板」證據再投資（需要機體
     庫存掃描＋地形適性/射程評估函數，成本高，不預先做）。

### 戰術層（內層 GOAP，2026-07-06 使用者定案）

戰鬥層不再是純反應式狀態機：`run()` 迴圈保留為感知/執行層（鎖、彈窗、
劇情、動畫等待——反應式雜務），決策核心改為 GOAP——行動以
precondition/effect/cost 描述、execute 綁既有 handler（handler 從
「決策者」降級為「執行器」），與戰略層共用同一套詞彙，行動才能動態插拔。

- **BattleState 統一盤面**：盤面知識不再散於 tacmap、controller 私有
  欄位與 ledger 之間——整併為單一世界狀態物件（各單位位置/數值 HP/EN/
  已行動/陣營/能力＋全域 predicates），感知寫入、planner 只讀、
  ledger 從它取快照。這就是知識分層裡的「戰鬥層」實體。
- **行動目錄（ActionCatalog）**：由「畫面掃描 ∪ 能力注入」生成
  （見設計原則 4）。戰略層在編成時現場讀取各出擊單位的能力
  （擊殺再動次數、技能……）寫入黑板 `TeamRoster`/`UnitCapability`；
  我方受養成影響，能力構成一律現場讀，roster cache 只記基礎 kit。
  能力**語意**是機制層 taxonomy（再動→行動經濟、EN 補給→續戰力、
  SUPPORT→夾擊……）；讀到 taxonomy 外的能力→轉錄進黑板與流水帳，
  但對 planner 惰性（不影響規劃、不會炸），等補類型後自然生效。
- **開戰驗證 action**：predicate `roster_verified`——確認實際出擊單位
  與編成設計一致，不符→現場重讀（畫面是權威）。
- **第三方友軍**：可控性視關卡而定（使用者 2026-07-06 確認），開戰時
  探測（嘗試選取該單位看行動選單是否開啟）。可控→其行動納入我方
  ActionCatalog；不可控→進世界狀態，其預期削血影響我方目標選擇。
- **啟動順序**（2026-07-06 使用者定案）：「選取單位」本身建為
  catalog 裡的 action；v1 planner 仍按固定順序（第一張卡），單機
  鏈式規劃驗證後再上跨機順序規劃（讓別機先削血、有雙再動的機收頭）。
- **單位啟動內鏈式規劃**：擊殺再動使規劃深度 = 1 + 剩餘再動次數，
  限定在單一單位的一次啟動內（≤3 步，窮舉即可）。`Attack(t)` 的
  effect 在 `預期傷害 ≥ HP(t)` 時為 `dead(t) ∧ 再動-1 ∧ 可再行動`，
  有再動在手時優先鎖定打得死的目標以串出多動。
- **擊殺判定用實際數值 HP**（使用者 2026-07-06 定案）：HP 弧比例會
  捏錯傷害，必須點進單位讀實際血量。黑板以「面板初值 − 實測傷害」
  追蹤數值 HP，執行擊殺鏈前點開目標重驗。弧比例只用於掃描與敵我分類。
- **戰鬥流水帳**：戰鬥中持續累積的觀測——各單位位置與 HP、
  每次交戰的（攻擊方、武器、預期傷害、實際傷害）、敵方每回合的移動與
  目標選擇、EN 耗盡等事件。終戰封存進 process 黑板，供同一 process 的
  下一次嘗試與戰後歸因使用。
- **評估策略分級**（依序實作）：
  1. 1-ply 貪婪：讀遊戲顯示的預期傷害/命中選武器×目標——遊戲本身就是
     forward model，零資料成本。
  2. 啟動內鏈式規劃（見上）：仍只用遊戲顯示的數字，零模型成本。
  3. 1.5-ply 站位安全：回合終點位置以威脅格＋黑板裡的敵方實測傷害評估
     「站這裡下回合會挨多少打」。
  4. 深搜尋（expectiminimax，2026-07-06 使用者定案追求關卡最佳解）：
     見下節「戰鬥模擬器與 expectiminimax」。

### 戰鬥模擬器與 expectiminimax（2026-07-06 使用者定案）

目標：**求關卡內最佳解**，決策必須超越 depth-1，後端模擬省不了。
模擬器是機制、代入數值是內容：`step(state, action) → state` 與公式
（docs/combat-formulas.md）寫進程式；機體/駕駛/武裝數值走資料綁定
（畫面讀取或 cache，之後由 LLM 讀數據或人工建檔）。當下這一步的命中/
傷害以遊戲攻擊預測 UI 顯示值為權威，公式只推演樹內假想局面。

**收錄機制（使用者逐項確認的遊戲規則）**：

- **相位制回合**：我方→第三方→敵方，該勢力全單位行動完自動輪換，
  回合數遞增。搜尋深度以相位為單位，葉節點評估放相位邊界。
- **擊殺再動**：擊殺後清除 acted、消耗一個 charge；**charge 於自己
  相位開始時重置**（跨回合搜尋須恢復），BattleState 上為動態欄位。
  **敵人同樣有再動與支援**，部分敵人的特殊能力會大幅提高動作數——
  屬內容，從敵方面板/cache 讀。
- **防禦應對是我方 max 節點**：敵方相位攻擊我方時，遊戲讓我方選
  {閃避（提高迴避率）、防禦（0.8×/盾 0.6× 減傷）、反擊（武器可選、
  受射程限制）}，之後補支援防禦/支援反擊分支——這是演算法要最佳化的
  決策，不是相位的附屬。節點歸屬看「當下誰做決定」，不看相位。
  執行層需新增敵方回合應對彈窗的感知與 handler（現行 controller
  對敵方相位純旁觀）。
- **格子移動與路徑封鎖**：地圖是格子；**敵方單位擋我方路徑，我方單位
  不擋自己人**（對稱地，我方擋敵方）。支援防禦距離＝該單位移動距離。
  站位因此可以：卡住敵方行動路徑、把敵方兩機隔到支援防禦距離外拆掉
  防禦網——保真度分兩階：v0 位置＋距離＋射程帶（支援防禦近似為
  「範圍內有支援者即觸發」）；v1 加格子可達性與碰撞，讓拆支援站位
  能被搜尋自己發現。
- **支援次數**：同再動，自己相位開始時重置。
- **chance 節點**：命中率（擊殺鏈對命中方差極敏感——85% 的「必殺」
  有 15% 讓整條再動鏈斷掉）；暴擊之後補。
- **敵方節點可插拔**（使用者定案）：預設**策略模型**（v0 最粗啟發式
  如打最近目標，之後由對戰經驗手動調整），介面保留 **min**（最壞
  情況）作保守驗證。

**搜尋定義（「關卡最佳解」的工程語意，使用者確認方向）**：

- anytime 迭代加深 expectiminimax＋置換表＋Star1/Star2 剪枝，
  搜到終局或預算耗盡。手遊無出手時限，單步可給數十秒至數分鐘；
  離線解全關＝同一 solver 加大預算，架構不變。
- 戰鬥後期樹自然塌縮，殘局自動達到精確全視野；開局為深視野＋
  葉節點評估（雙方剩餘 HP 加權＋擊墜數起步）。
- 每次實機行動後感知回寫、滾動重規劃，開局近似誤差被逐步修正。
- 既有啟動內鏈式規劃（≤3 步窮舉）成為 solver 在「單一啟動」子樹
  的特例；BattleState 需支援便宜 clone 供節點展開。

## 執行期敵情獲取機制

戰鬥開局把敵人資訊**全部**寫進當局黑板，之後每回合更新（使用者 2026-07-05
定案的流程）：

1. **開局全圖掃描**：平移鏡頭掃過整張地圖，每次平移用相位相關實測鏡頭
   位移（不假設滑多遠動多遠），HP 弧的螢幕座標＋鏡頭偏移＝世界座標，
   建立敵/我/第三方的戰術地圖（`battle/tacmap.py`）。
2. **逐一點開敵方資訊**：每個敵人的攻擊招式範圍與特殊能力都在敵方資訊
   面板裡——面板**結構**標定一次（機制），**內容**每關現讀，附掛到該敵人
   的黑板條目上。招式範圍也可從點選敵人時遊戲畫出的範圍疊加層以顏色
   遮罩通用辨識。
3. **每回合更新**：我方回合開頭重掃位置（單位會動，位置以當回合為準）；
   能力/招式屬於機體不會變，開局讀一次即可。
4. 交戰實測傷害即時記入黑板，回合內就能用於站位評估；同型敵機共享估計。

鏡頭錨定：選取單位時遊戲會把鏡頭跳到該單位（不可量測的跳躍），移動決策
前用「可見 HP 弧星座 vs 黑板世界座標」的平移匹配找回鏡頭偏移，
畫面外的敵人因此也能成為移動目標。

## 內容 cache → 關卡定義檔（2026-07-14 演進）

感知記憶化 cache（2026-07-05 定案的 sig-keyed kit 快取）已定案演進為
**關卡定義檔**：layout（uid/格位/機種/駕駛/數值）＋conditions（勝利/
失敗條件）＋events（觸發→盤面變化），後端 uid 為唯一身分、名牌 sig
降為關聯證據。需求全文見
[stage-definition-requirements.md](stage-definition-requirements.md)；
舊三機制設計全文見 [archive.md](archive.md)。不變的原則：**畫面永遠
是權威**（開局校驗、不符整關過期退回現場全讀）、我方實數受養成影響
一律現場同步、cache 檔記遊戲版號。

## 戰後歸因（觀測式，非模擬式）

流水帳證據 → 診斷 → 補強方向的對映，關鍵是能分辨
「該花資源養成」vs「該修我們的程式」：

| 證據 | 診斷 | 補強 |
|---|---|---|
| 前兩回合我方大量陣亡、擊殺少 | 純數值差距 | farm＋強化（戰略層） |
| 打得中但敵 HP 弧幾乎不掉 | 火力不足 | 武器強化（戰略層） |
| 多次 EN 耗盡只能待機 | 戰術缺陷 | 行動掃描＋技能使用（改程式） |
| 同一單位總是最先死 | 編成短板 | 編成調整（戰略層 v2） |
| 回合耗盡未分勝負 | 機動/索敵問題 | 改程式 |
| 有再動能力卻未串出擊殺鏈 | 戰術缺陷 | planner 能力運用（改程式） |
